# 通用 Skill 质量检查清单

创建或审查一个 Agent Skill 时，请逐条检查以下三个维度。

---

## 维度一：结构完整性

| # | 检查项 | 达标标准 | ✅/❌ |
|---|---|---|---|
| 1 | **目录结构** | 顶层只有 `SKILL.md` + `examples/` + `resources/`，无多余目录 | |
| 2 | **YAML Frontmatter** | 包含 `name`（与文件夹名一致）和 `description`（说明做什么 + 什么时候用） | |
| 3 | **8 大模块完整性** | SKILL.md 包含全部 8 个模块：Frontmatter → 角色 → 输入 → 工作流 → 变体 → 约束 → 自检 → 链接 | |
| 4 | **链接可达性** | 快速参考链接中的每个路径都指向真实存在的文件 | |
| 5 | **文件覆盖** | examples/ 和 resources/ 中的每个文件都在 SKILL.md 中被引用 | |

## 维度二：内容质量

| # | 检查项 | 达标标准 | ✅/❌ |
|---|---|---|---|
| 6 | **角色具体性** | 角色定位是否有具体领域身份？是否有明确的成功指标？是否避免了"通用助手"式表述？ | |
| 7 | **工作流原子性** | 每个步骤是否是可独立执行的原子操作？步骤间是否有明确的数据流转？ | |
| 8 | **输出格式明确性** | 是否用代码块提供了标准输出模板？Agent 是否能照着模板直接填充？ | |
| 9 | **案例质量** | examples/ 是否包含 ≥ 2 个差异化案例？每个案例是否有完整的输入→输出→策略标注？ | |
| 10 | **约束可执行性** | 约束和禁忌是否具体到可量化？（如："禁止超过 150 词" 而非 "不要写太长"） | |
| 11 | **自检清单有效性** | 质量自检是否有 ≥ 6 项？达标标准是否具体可量化？ | |

## 维度三：可用性

| # | 检查项 | 达标标准 | ✅/❌ |
|---|---|---|---|
| 12 | **SKILL.md 长度** | 总长度在 100-300 行之间（太短=信息不足，太长=注意力稀释） | |
| 13 | **信息密度** | 无冗余废话、无重复段落、无空洞的形容词堆砌 | |
| 14 | **禁用词** | 全文无 AI 机械词汇（Elevate, Unleash, Revolutionary, Game-changing, Streamline） | |
| 15 | **可读性** | 使用表格、代码块、分点列表等排版手段，而非大段纯文本 | |
| 16 | **Skill 链路兼容** | 如果有上下游 Skill，输入输出格式是否与上下游兼容？ | |

---

## 使用方法

1. 创建完 Skill 后，打印此清单
2. 逐条检查，在最右列标记 ✅ 或 ❌
3. 任何 ❌ 项必须修正后再交付
4. 全部 ✅ 后，Skill 即可投入使用
